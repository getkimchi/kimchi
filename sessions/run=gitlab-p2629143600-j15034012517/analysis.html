5. Agent Behavior Observations

5.1 Cache is effectively off for the orchestrator

Every minimax-m3 orchestrator row in the summary has cache_read: 0, cache_write: 0. Only kimi-k2.x sub-agents register any cache activity (e.g. cobol-modernization a2: 10k cache_read on k2.7; extract-elf a3: 62k; merge-diff-arc-agi: 165k; mteb-leaderboard: 367k). On a benchmark where the orchestrator's input bill alone averages 1.2M tokens per trial, eliminating prompt-prefix caching on minimax-m3 is leaving real performance and cost on the table.

5.2 Sub-agent selection is unstable across attempts

For the same task, retry attempts pick different sub-agent models (or none). Examples:

  • build-cython-ext: a1 → m2.7; a2 → none (orchestrator solo); a3 → k2.6 + m2.7.
  • cobol-modernization: a1 → m2.7; a2 → m2.7 + k2.7; a3 → none (aborts at planning).
  • pytorch-model-recovery: a1 → m2.7; a2 → none; a3 → k2.6.

This is a multi-model agent without a deterministic routing policy. For a benchmark we want reproducibility; the current behaviour makes "did sub-agent X help?" hard to attribute.

5.3 Heavy delegation does not correlate with success

Trials with the most Agent spawns:

  • merge-diff-arc-agi-task: 11 Agent calls → PASS
  • cobol-modernization a2: 11 → TIMEOUT
  • build-cython-ext a3: 10 → TIMEOUT
  • bn-fit-modify: 8 → FAIL
  • mteb-leaderboard: 4 → PASS

The pattern is bimodal: a few passes work despite high fan-out (merge-diff-arc-agi-task), but the failure cases cluster at the same fan-out level. The branch under test (LLM-fix-overbroad-subagent-prompts) is explicitly trying to address this, so the run is measuring exactly this dimension.

5.4 The ferment lifecycle does not enforce end-to-end verification

Across the 18 trials the orchestrator reaches complete_ferment in many of them, but the verifier still rejects the artifact (extract-elf a2 is the clearest example). Each ferment step is "completed" individually; the harness appears to have no gate that says "before you complete the ferment, run the produced script end-to-end and inspect the output". The run-time data suggests this would catch at least the extract-elf class of failures.

5.5 Cost asymmetry between pass and fail is small; cost asymmetry between timeout and within-budget is large

Within-budget runs average ~17m; timeout runs are all pinned to ~45m by the budget. So timeouts are not a function of model speed — they are a function of the work plan being too large for 2700s. Increasing the budget would help the wall-clock failures linearly; increasing plan quality would help them super-linearly.

6. Recommendations

For the agent

  1. Fix the planning-phase abort path (cobol-modernization a3). When the orchestrator's first turn produces no tool call under the ferment planner profile, the run aborts after a single round. Either make the planner profile retry, or escalate to a follow-up turn, instead of letting the trial die. Investigate whether the "Toolset follows the ferment lifecycle" toolset restriction is the proximate cause of the early termination.
  2. Add an end-to-end verification step before complete_ferment. Even a single mandatory "run the produced artifact and check the verifier's output paths" step would have caught the empty-JSON extract-elf a2 case and likely the build-cython-ext empty-import cases.
  3. Investigate prompt-prefix caching on minimax-m3. Zero cache utilisation across all orchestrator runs is the largest single efficiency loss in the dataset.
  4. Make sub-agent selection deterministic and per-task-profile. The current non-determinism (different sub-agent model per attempt, sometimes none) makes evaluation noisy. Even a static mapping "this task class → this sub-agent model" would help.
  5. Reduce default per-step budgets where the work is mechanical. extract-elf a2 ran 41 bash calls under one sub-agent without producing a non-empty output. A short "check after N steps that the artifact file is valid" guard inside the worker would have surfaced the problem far earlier.

For the benchmark harness

  1. Re-run with timeout_multiplier: 5 on the three tasks that failed every attempt (build-cython-ext, cobol-modernization, extract-elf) to separate "needs more time" from "needs different model".
  2. Surface a per-attempt sub-agent model trace in summary.json. Today the trail mixes orchestrator and sub-agent token totals per trial but not per attempt; knowing which sub-agent model was used in each retry is needed for the multi-model routing analysis.
  3. Re-classify scored_fail trials that hit the budget cap as agent_timeout in summary.totals. Currently, a trial that hits 2700s and then verifier-0s is counted as failed_quality, but it was the budget that killed it. Several of the 6 "scored_fail" rows in totals are functionally timeouts.
  4. Persist session JSONLs in a stable location. The find tool's default .gitignore handling hid every main.jsonl under agent/sessions/ from filesystem walks; analysts need to read them by path. Either lift them out of .gitignore territory, or symlink them into the analysis directory.
  5. Plan for is_complete: false. The summary explicitly flags that chunk-0 exhausted retries, so the 18 reported trials are not the entire intended workload. Re-running the exhausted chunk should be the first follow-up before drawing final conclusions.

For the model evaluation

  1. Isolate minimax-m3 from minimax-m2.7 effects. In multi-model mode the orchestrator and sub-agent contribute to the same trial score. To attribute capability, run a single-model ablation (orchestrator = sub-agent = minimax-m3) on the same task list and compare.
  2. Quantify the "sub-agent helps" effect on the kimi-k2.x branch. 4/7 = 57% pass rate when k2.x runs vs 3/10 = 30% when only m2.7 runs. A targeted ablation — same task, kimi-k2.x sub-agent vs m2.7 sub-agent — would give a clean signal.
  3. Pre-classify tasks by expected difficulty (e.g. build/compile vs file-transformation vs git-history recovery) and report pass-rate per bucket. The current aggregate 33% hides that file-transformation tasks (fix-git, fix-code-vulnerability) are 100% on first try while build tasks (build-cython-ext) are 0%.
