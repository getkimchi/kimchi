Failure deep-dives

1. bn-fit-modify — Bayesian Network DAG recovery & intervention

The task requires recovering a 6-edge DAG over 5 continuous variables (U, Y, R, D, M) from a 10k-row sample, fitting a Bayesian Network, performing do(Y=0), and sampling 10k new points. The task explicitly states: "the alphabetically-first letter is the child of the node that comes second" when edge direction is ambiguous.

What failed (3 tests):

  • test_learned_dag_structure — assertion failed because the learned edge set contained ('D','Y') instead of the expected ('Y','D'). That is, the agent inverted one edge direction in the DAG. The reference expected {('U','M'), ('U','Y'), ('U','D'), ('U','R'), ('Y','D'), ('R','M')}; the agent's output included the same six edges but with Y and D swapped on one edge.
  • test_intervened__data_structure — the intervened DAG consequently inherited the wrong parent for D, breaking the structural comparison.
  • test_sampled_data — the KS test for D's marginal distribution returned p = 0.0 against the expected Gaussian parameters (μ = -12.30 + 0.5496·50.48, σ = √(0.5496²·10.69² + 14.09²)). A wrong parent for D means the fitted linear-Gaussian coefficients are wrong, so the forward-sampled distribution diverges from the expected one.

Root cause. A single edge-direction error (one arrow flipped between D and Y) cascaded into structural and distributional failures. The agent's plan even called this out explicitly: "Apply alphabetical tiebreaking for undirected edges (first-letter-is-child)", but the post-constraint PC/HillClimbSearch output disagreed with the alphabetical rule on at least one edge. PC-style algorithms can leave edges undirected when conditional independence is symmetric; the alphabetical tiebreaker is supposed to settle those, but the agent apparently either ran an algorithm that already produced a directed graph it then trusted, or applied the rule inconsistently.

Session telemetry. 41 planner LLM rounds + 58 worker rounds (k2.7 subagent). 1.35M + 0.35M input tokens. Duration 22m30s. The verification error stored in the summary is the unhelpful "Traceback (most recent call last):" snippet, indicating a traceback was emitted but the captured message was truncated.

2. extract-elf — ELF symbol extraction to JSON

The task asks the agent to write extract.js that, given a compiled ELF binary, outputs a JSON file mapping memory addresses to values consistent with a reference extractor.

What failed (1 test):

  • test_output_matches_reference — the agent's extract.js produced an out.json that contained 0.00% of the reference keys. The verifier requires ≥ 75% overlap and zero inconsistent values, so the agent missed the bar entirely.

Root cause. The summary records: "Agent aborted in 240.2s (13 tool uses, 30.4k token) (aborted — wall-clock duration limit exceeded, output may be incomplete)." The agent hit a hard wall-clock limit (240 s) on the subagent that was implementing the JS extractor. With only 13 tool calls and ~30k output tokens, the worker never produced a working extract.js. The verifier still found a file present and syntactically valid (so test_extract_js_exists passed), but its output was essentially empty or wrong-shaped.

Session telemetry. 14 planner LLM rounds + 12 worker rounds (m2.7 subagent). 0.47M + 0.05M input tokens. Total duration 15m22s; the worker aborted ~4 minutes in.

Agent behavior & tool usage observations

Ferment lifecycle

The agent runs under a ferment-oneshot mode: it must scope_ferment, then call activate_ferment_phase → start_ferment_step → spawn an Agent worker → complete_ferment_step, finally complete_ferment. Every turn the orchestrator (m3) ends with either a lifecycle tool call or an Agent spawn. The prompts the orchestrator writes are unusually prescriptive — they tell the worker its persona, budget, plan structure, and what to do if the budget is exhausted.

Model mixing

The planner model minimax-m3 handles scoping, lifecycle bookkeeping, and direct file edits when no subagent is needed. It also sees many more tool calls on the simple tasks (build-cython-ext: 149 bash calls; fix-code-vulnerability: 71 rounds total). For substantive implementation steps, it dispatches an Agent worker using either kimi-k2.7 (bn-fit-modify, cobol, fix-git, mteb-leaderboard) or minimax-m2.7 (extract-elf, hf-model-inference, merge-diff-arc-agi-task, pytorch-model-recovery). The choice appears to be the planner's, not fixed by task type.

Worker budgets (max_turns / max_duration)

Workers were dispatched with budgets sized to step complexity (e.g. max_turns=15, max_duration=300 for read-only exploration; larger budgets for implementation). When a worker exhausts its budget, the orchestrator is instructed to call complete_ferment_step with whatever it produced and spawn a scoped follow-up rather than re-running the same broad task with a bigger budget. In extract-elf, this is exactly what went wrong: the wall-clock duration limit killed the worker before it converged, and no follow-up was spawned within the remaining budget.

Actionable recommendations

For the agent

  1. Add a structural self-check after DAG learning. Before writing learned_dag.csv, the worker should (a) re-read the task's tiebreaking rule, (b) enumerate the undirected edges the algorithm produced, (c) apply the rule, and (d) verify the resulting edge set against the count and no-parents-of-U constraints. This would have caught the single flipped Y/D edge in bn-fit-modify.
  2. Sanity-check fitted parameters before sampling. Once the linear-Gaussian BN is fit, print model.cpds and check that coefficients for D match the expected sign/magnitude from the original problem. A KS test on a calibration subset is cheap and would have surfaced the wrong-parent bug before the verifier ran.
  3. Increase or stage the wall-clock budget for ELF-class tasks. The 240 s cap on the implementation worker killed extract-elf at 13 tool calls. Either raise the default for binary-parsing subtasks to 600 s, or break the extractor into a write → smoke-test → iterate loop with the orchestrator pulling intermediate results back via get_subagent_result before the worker hits its budget.
  4. Avoid model-name typos in worker dispatches. The haiku mis-spawn in bn-fit-modify was recoverable only because the harness returned a useful available models list in the error. Make the orchestrator prompt include the actual model registry, or have the planner confirm model selection against the list returned by the first failed spawn.
  5. Surface verifier messages clearly in the summary. The bn-fit-modify summary stored the truncated string "Traceback (most recent call last):" as the error message. The real pytest traceback is in verifier/ctrf.json but the classification step didn't propagate it. Include the failing test name + first line of the assertion error in the summary.

For the benchmark

  1. Tighten the DAG ambiguity rule language. bn-fit-modify's instruction "edges not involving U for which directionality is not clear, in this case you may assume that the node labeled with the letter that comes first in the alphabet is the child of the node that comes second" allows two readings: alphabetical by source (the alphabetically-first letter is the child of the alphabetically-second) and alphabetical by undirected edge ordering. A small worked example would remove the ambiguity and let the verifier catch mis-reads even when the algorithm's directed output disagrees with the rule.
  2. Add an example out.json shape to extract-elf. The current verifier expects "memory addresses → values" but the precise schema (key type, value type, expected keys count) is only visible from the reference. Providing a small example in the task prompt would let the agent converge in fewer tool calls and reduce the chance of a 0%-overlap output.
  3. Reclassify "Agent aborted" as a distinct verdict. extract-elf failed because of an internal wall-clock cap, not because the agent produced a wrong answer. Treating it alongside wrong-answer failures obscures the real signal. Add an aborted outcome category distinct from failed_quality.
  4. Verify that out.json schema matches reference.json for the extract-elf task. A pre-check that the student's output JSON parses and shares the same top-level type with the reference would localize the failure to "we wrote the wrong schema" rather than "0% of the expected values overlap" — a more actionable error for the agent.
