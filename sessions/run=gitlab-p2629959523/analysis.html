The agent configuration under test (multi-mode-ferment with primary model kimi-k2.7 and orchestrator minimax-m3) achieved a 6/10 pass rate against terminal-bench-2. The four failures cluster into two distinct failure modes:

  • Orchestrator premature-exit (2 trials): cobol-modernization and fix-code-vulnerability each used exactly one LLM round, made zero tool calls, and exited. The orchestrator produced detailed thinking about how it would call scope_ferment but never actually issued any tool call.

3. Failure Mode Taxonomy

The four failed trials cluster cleanly into three categories. Below is the failure breakdown with explicit evidence drawn from the session logs.

3.1 Category A — Orchestrator premature-exit (silent zero-tool termination)

Affected: cobol-modernization, fix-code-vulnerability Both trials ended after a single LLM round with toolCalls: []. The orchestrator produced a long internal thinking block describing how it would call scope_ferment, followed by a short text reply ("I'll execute this one-shot ferment…"), and then the session terminated. No scope_ferment, no activate_ferment_phase, no work of any kind was performed. Both trials ended in under 100 seconds, both with verifier reward 0.0.

Evidence from cobol-modernization/main.jsonl (assistant message truncated):

{
  "role": "assistant",
  "stopReason": "stop",
  "content": [
    { "type": "thinking", "thinking": "Thinking: The user wants me to execute a one-shot ferment…" },
    { "type": "text",     "text":     "I'll execute this one-shot ferment by first scoping the plan, then running through each phase." }
  ],
  "toolCalls": []
}

Both files end with a prompt-summary custom event noting "1m 19s" / "58.9s" total — and the verdict's error.message is just the tail of the verifier's pytest output (=================================== FAILURES =================================== or the bottle.py test instructions), confirming the agent never produced any file output for the verifier to score. This is the single highest-impact failure pattern in the run: two out of ten failures (20% of all trials, 50% of all failures) are caused entirely by the orchestrator deciding to stop without calling any tool.

Why this happens (hypothesis)

  • The system prompt explicitly orders the first turn: "Call scope_ferment as your VERY FIRST tool call on this turn." Despite that, the orchestrator's text reply reads "I'll execute this one-shot ferment…" — i.e. it narrates intent rather than executing it.
  • The session shows no error, no tool-call refusal, and no guard-steered nudge. The model simply chose stopReason: "stop" after writing one sentence. There is no evidence in the log that the agent harness retried or detected the missed tool call.
  • Both affected trials contain a one-shot ferment preamble and are unusually long (multi-thousand-token) user prompts. The orchestrator may have treated the long prompt as a planning task and produced a thinking-only response.

3.2 Category B — Hard timeout on heavy multi-phase work

Affected: build-cython-ext Outcome: AgentTimeoutError: Agent execution timed out after 2700.0 seconds. Verifier reward 0.0.

Evidence: the trial consumed 88 LLM rounds on the worker (kimi-k2.6) and 28 rounds on the orchestrator (minimax-m3), totalling 4.29M input tokens — the highest token usage of the run. Session timestamps show the agent making real progress (cloned pyknotid 0.5.3, surveyed NumPy 1.x deprecated APIs across the tree, applied targeted np.int → int, n.double → n.float64, dtype=bool, distutils → setuptools, etc. patches across 12 .py files and 2 .pyx files) before the 2700-second wall clock fired. The session was cancelled mid-phase-3 (build/install/verify).

What worked:

  • Plan was sound — 3 phases with verified gates and explicit verify commands.
  • Patches were applied accurately (e.g. ccomplexity.pyx np.int → int, coctree.pyx n.double → n.float64).
  • Worker reports (e.g. the step-2-1-patches.md audit notes) show clean, minimal diffs.

What did not work:

  • The agent spent too much round budget on Phase 2 (12 .py files + 2 .pyx files patched across 3 separate complete-ferment-step calls before reaching Phase 3).
  • The plan did not anticipate that pip install . from Cython source might fail and need iterative debugging — Phase 3 step 1 alone (build) was the obvious place to eat the rest of the budget.
  • With a 3.0× timeout multiplier and no override, this trial had effectively 45 minutes of wall clock. Trial ended ~5s before the next checkpoint.

3.3 Category C — Research succeeds, artefact mismatch

Affected: mteb-leaderboard Outcome: scored_fail with verifier reward: 0.0. Trial error.message contains the literal token startup_duration_timeout: 1h, suggesting the verifier hit its 1-hour timeout while waiting for the agent environment to come up.

The orchestrator and worker collaborated through 61 rounds (35 + 26) and 1.73M input tokens of web research. The work itself was largely correct:

  • Correctly identified the URL https://huggingface.co/spaces/mteb/leaderboard via the mteb/MTEB README and the HF Space.
  • Cross-referenced the HKasar1239/scandinavian-embedding-toolkit card and the TTC blog to land on the model id jealk/TTC-L2V-supervised-2.
  • Wrote /app/.kimchi/docs/research.md with the URL, model id on its own line, and four pieces of corroborating evidence.

Failure evidence:

  • The orchestrator initially tried to spawn a sub-agent with model: "sonnet" — a model that does not exist in the configured provider. The Agent tool returned a "Model not found" listing of valid IDs (kimi-k2.6, kimi-k2.7, minimax-m2.7, minimax-m3, plus glm-5.2-fp8 and nemotron-3-ultra-fp4). The orchestrator retried with kimi-k2.7.
  • The first sub-agent (researcher) hit its max_turns=20 budget in 149.3s without producing output. The orchestrator then decided to do the research itself with web_search/web_fetch.
  • The session shows the model id was captured in research notes, but the trail of the trial does not show a final write of /app/result.txt in the main session log within the visible window. The verifier scored 0.0, indicating the file was either absent, empty, or did not contain the expected org/model id format.
  • The summary field carries the timestamp startup_duration_timeout: 1h as the verifier error — implying the verifier pipeline itself timed out before it could run. This may also be a noisy error.message tag from classify.py that captured a partial string.
Cross-cutting observation: categories A and C both involve the orchestrator failing to converge on the actual deliverable file the verifier checks (program.cbl → program.py for cobol-modernization; report.jsonl for fix-code-vulnerability; result.txt for mteb-leaderboard). The harness's tooling did not detect the missing artefact. A "deliverable-exists" smoke check before the verifier runs would have caught categories A and likely C.

5. Agent Behaviour Observations

5.1 Multi-model architecture in practice

Every trial used multi-model: true and ferment-oneshot: true in the agent config. In practice this resolved to:

  • Orchestrator: always minimax-m3 (1 LLM round for category-A failures; 18–46 rounds for everything else).
  • Workers: kimi-k2.7 (3 passes), kimi-k2.6 (4 passes + 1 fail), minimax-m2.7 (1 pass on merge-diff-arc-agi-task).
  • Worker subagent types observed: Researcher (mteb-leaderboard step 1), General-Purpose (clone/init/pyx-patch steps), Builder (BN dataset inspection).

The split between planning rounds (orchestrator) and execution rounds (worker) was roughly 1:1 for most passes (e.g. bn-fit-modify: 46/48, pytorch-model-recovery: 39/41, extract-elf: 25/31). This is consistent with the "planner profile → subagent with full toolset" pattern documented in the agent's lifecycle tools.

5.2 Ferment lifecycle adherence

Passing trials all followed the canonical lifecycle:

scope_ferment
  → activate_ferment_phase (×N phases)
    → start_ferment_step
      → Agent(subagent_type=..., max_turns=..., max_duration=...)
      → complete_ferment_step
    → next step…
  → complete_ferment_phase
  → next phase…
→ complete_ferment

The lifecycle tool is strict about gate arrays (the prompt says "The schema hard-rejects calls missing this array"), and all passing trials emitted well-formed gates: [{id, verdict, rationale, evidence}] arrays. Workers received explicit Worker Context blocks with the phase goal, step verify command, success criteria, and the audit-notes directory.

5.3 Tool usage patterns

  • bash + read + grep are dominant for the worker (e.g. build-cython-ext: 143 bash, 58 read, 15 grep calls).
  • Agent is heavily used by the orchestrator as the delegation primitive (3–6 calls per trial in the pass set).
  • Ferment lifecycle tools (scope_ferment, activate_ferment_phase, start_ferment_step, complete_ferment_step, complete_ferment_phase, complete_ferment) are 8–13 calls per passing trial — every phase and step transition is named explicitly.
  • web_search / web_fetch only appear in mteb-leaderboard, as expected.
  • create_todos was used once (build-cython-ext).

5.4 Behaviour guardrails

Every session log includes a behaviour_loaded / behaviour_session_summary event noting the active guards:

  • bound-tool-output — observed: 0, violated: 0 across all sessions. Guard appears to have steered but never tripped.
  • re-read-before-edit — observed: 0, violated: 0. The guard was loaded but not invoked during the runs examined.
  • git-hygiene — loaded on session start for tasks involving git (fix-git, merge-diff-arc-agi-task).

The bash-tool-guard-steer custom event appears throughout build-cython-ext's main.jsonl, indicating the guard nudged the model away from head/tail-style bash use toward read. This is positive — but only the surviving trials benefited from it.

5.5 Subagent budget management

The system prompt mandates max_turns and max_duration on every Agent call and warns "When the worker exhausts its budget, call complete_ferment_step with whatever it produced and spawn a scoped follow-up". In mteb-leaderboard the first subagent hit max_turns=20 in 26 actual turns and 149.3s, returning "No output." — exactly the failure mode the prompt anticipates. The orchestrator recovered by re-scoping and continuing the work in the main loop. This is the documented recovery path in action, but it cost enough rounds that the final result.txt write did not complete in time.

5.6 Model id hallucination

mteb-leaderboard's first subagent dispatch used model: "sonnet" — not in the configured registry. The Agent tool refused and listed valid IDs. The orchestrator self-corrected. This is a one-off hallucination, not a recurring pattern, but worth noting because none of the other trials exposed it (they all used model ids that existed).

6.1 Token efficiency

  • Failed "premature exit" trials consumed just 24K–25K input tokens — the cheapest signal in the run. The cost of premature exit is essentially zero; the cost of not detecting it is two missed scores.
  • Passes averaged ~1.4M input tokens per trial. The cheapest pass (hf-model-inference) used 615K. The most expensive pass (bn-fit-modify) used 2.37M.

6.2 Round count vs duration

Round counts scale roughly linearly with token cost, but duration does not — the orchestration overhead per round varies by 3–10× depending on whether subagents are involved. Build-cython-ext averaged 11.8 rounds/minute in its final stretch; cobol-modernization did the work in a single 79-second round. The cost asymmetry suggests the orchestrator should detect the "this is going to need many rounds" signal earlier in long-horizon tasks and either narrow scope or break the work into more parallel subagents.

8. Recommendations

8.1 Fix the orchestrator's premature-exit pattern (highest impact)

  1. Add a no-tool-call guardrail at the orchestrator layer. If the orchestrator finishes its first turn without any tool call, the harness should inject a follow-up message like "Your task requires you to call scope_ferment first. Issue the tool call now." before the session is declared complete. The two category-A failures would both have been caught by this single rule.
  2. Detect "narrative-only" assistant messages. When the assistant text contains first-person future-tense narration ("I'll do X", "Let me start by Y") without any accompanying tool_calls, nudge the model or refuse to end the turn.
  3. Make the system prompt harder to misinterpret. The current prompt's "Call scope_ferment as your VERY FIRST tool call on this turn" is a soft instruction; consider turning it into a structured first-turn contract: { "tool": "scope_ferment", "required": true } that the harness enforces before accepting a turn without a tool call.
  4. Add a post-orchestrator smoke check before the verifier. For each task, the harness knows the deliverable paths (e.g. /app/program.py, /app/result.txt, /app/report.jsonl). Run a cheap test -s on each before the verifier fires and force a re-run of the final phase if the file is missing or empty.

8.2 Improve long-horizon budget management

  1. Right-size per-step budgets based on step complexity. The system prompt already prescribes 20–30/300s, 40–60/600s, 80/900s bands. The trials examined used 20/300 and 40/600 throughout; build-cython-ext's later steps probably needed 80/900.
  2. Collapse tightly-coupled steps into a single subagent. bn-fit-modify and pytorch-model-recovery got 5+ round-trips per phase that could have been 1–2. The current instruction explicitly warns against batching because "the steps are tightly coupled", but it should warn against the opposite — fragmenting a phase into N subagents when they need shared state.
  3. Surface remaining budget to the orchestrator. If the orchestrator had a clear "Phase 3 of 3, ~12 minutes of budget remaining" indicator, it could skip non-essential verification and prioritise the final deliverable.
  4. For Cython/CMake-heavy tasks, plan a fast-fail build check early. build-cython-ext ran 12 file patches before checking whether the package would even import. A "compile-import sanity check after every 3 patches" rule would have surfaced deeper issues earlier or revealed that the patches were sufficient.

8.3 Reduce model-id hallucination

  1. Constrain the model parameter on Agent dispatches. Pass the list of valid models into the orchestrator's system context or have the harness validate model ids before invoking the subagent. The single hallucination of "sonnet" cost a full subagent round-trip.
  2. Surface the configured model registry in the prompt. The mteb-leaderboard failure shows the orchestrator guessed at model names. Listing "Available workers: kimchi-dev/kimi-k2.7, kimchi-dev/kimi-k2.6, kimchi-dev/minimax-m2.7, kimchi-dev/glm-5.2-fp8, kimchi-dev/nemotron-3-ultra-fp4" at the top of the planning prompt would prevent this.

8.4 Benchmark-side improvements

  1. Tighter timeout policy for simple tasks. cobol-modernization and fix-code-vulnerability each had 3.0× multiplier but exited in ~85 seconds — there is no signal in the timeout that they failed quickly. A "minimum-viable time" check (e.g. "agent must have made >10 tool calls within first 60s") would force re-runs.
  2. Improve classify.py error capture. The mteb-leaderboard error.message contains startup_duration_timeout: 1h, which looks like a stale string rather than a precise verdict. Consider storing a structured error category alongside the message.
  3. Add a "no-tool-call" verdict. Currently the two category-A failures are both scored_fail. A distinct premature_exit verdict would let dashboards track this pattern separately from "agent tried, didn't converge".

8.5 Things to keep doing

  • The ferment lifecycle with strict gate arrays worked. All 6 passes produced well-formed gates arrays with evidence pointers. This is good discipline.
  • The multi-model split (planner profile orchestrator + full-toolset subagents) produced clean separation of concerns. merge-diff-arc-agi-task with minimax-m2.7 workers performed comparably to kimi-k2.7-driven trials.
  • The audit-notes directory pattern (each ferment writes progress to .kimchi/ferments/<id>/docs/) produced high-signal recovery artifacts. This was particularly visible in build-cython-ext's step-2-1-patches.md and mteb-leaderboard's research.md.
  • The behaviour guards (bound-tool-output, re-read-before-edit) were observed but never violated in the runs sampled. The bash-tool-guard-steer nudges were effective at keeping workers on the dedicated read tool.
