Kimchi internals

How compaction works in the harness

Compaction is the mechanism that lets long sessions keep going past the model's context window. The core logic lives upstream in @earendil-works/pi-coding-agent; Kimchi mostly hooks into it and re-exposes the events through its Claude Code hook adapter.

3

Triggers

Manual /compact, reactive overflow recovery, and a proactive threshold check at the end of each turn. All three call the same internal _runAutoCompaction() path.

5

Steps

Prepare Hook Summarize Append Reload

Session manager walks entries backward, fires session_before_compact, asks the LLM for a structured summary, then appends a CompactionEntry and reloads the session.

2

Kimchi's role

Kimchi does not change when or how compaction happens. It registers a /compact slash-command entry and translates the upstream events into Claude Code PreCompact / PostCompact hook calls so user scripts can react.

Where it lives

Who owns the loop, who hooks it

Compaction is part of the upstream AgentSession turn loop. Kimchi extends the loop with extension factories and a hook adapter but does not reimplement the trigger or summarization code.

Concern Owner Source
Threshold check, cut-point search, LLM summary call Upstream pi-coding-agent node_modules/@earendil-works/pi-coding-agent/dist/core/compaction/compaction.js
Session entry append, reload, branching Upstream AgentSession node_modules/@earendil-works/pi-coding-agent/dist/core/agent-session.js (_runAutoCompaction)
Slash command registration Upstream Kimchi only advertises it via src/extensions/help.ts

Settings (reserveTokens, keepRecentTokens) Upstream reads ~/.pi/agent/settings.json or .pi/settings.json Kimchi does not override these defaults

Bottom line. Kimchi sits on the side of the loop. The decisions about when to compact and what the summary looks like are made by the upstream package. Kimchi's job is to keep its own UI, subagent bookkeeping, and Claude Code compatibility hooks consistent with whatever upstream decides.

Triggers

When does a session get compacted?

There are exactly three paths into compaction. They all converge on the same internal helper, AgentSession._runAutoCompaction(reason, willRetry).

1 В· Manual User types /compact (or /compact <instructions>) in the editor. Upstream runs the command and enters the same compaction path with customInstructions set.
2 В· Threshold After every turn, the session compares contextTokens > contextWindow вҲ’ reserveTokens. If true, runs _runAutoCompaction("threshold", false) before the next user prompt.
3 В· Overflow When the model returns a context-overflow error, the session removes the failing assistant message and runs _runAutoCompaction("overflow", true), then retries the turn.
Hook veto If an extension handler returns { cancel: true } from session_before_compact, the run is aborted and the original turn / retry proceeds without compaction.
Hook override Extensions may return a fully-formed { compaction: { summary, firstKeptEntryId, tokensBefore, details } } to skip the LLM summary call and use their own summary instead.

Threshold math (verbatim from upstream)

// node_modules/@earendil-works/pi-coding-agent/dist/core/compaction/compaction.js
export function shouldCompact(contextTokens, contextWindow, settings) {
    if (!settings.enabled) return false;
    return contextTokens > contextWindow - settings.reserveTokens;
}
Setting Default Effect Configured in
compaction.enabled true Master switch. false disables both proactive threshold and overflow recovery; manual /compact still works. ~/.pi/agent/settings.json or .pi/settings.json
compaction.reserveTokens 16384 Headroom kept free for the LLM's next response. Trigger fires when contextWindow вҲ’ reserveTokens is exceeded. Same as above
compaction.keepRecentTokens 20000 How much of the recent tail is preserved verbatim. Anything older gets summarized. Same as above

Kimchi does not override any of these. grep -r "reserveTokens\|keepRecentTokens\|compaction\.enabled" across src/ returns no hits. To tune compaction, edit the upstream settings file directly вҖ” Kimchi will pick up the same value.

Algorithm

How the compaction is done

The implementation lives in @earendil-works/pi-coding-agent/dist/core/compaction/compaction.js (plus utils.js for serialization and agent-session.js for orchestration). The pipeline has five stages.

Prepare prepareCompaction(pathEntries, settings) walks entries from newest to oldest, accumulating estimated tokens until keepRecentTokens is reached. The cut lands on a valid boundary (user, assistant, bash execution, or custom message вҖ” never a tool result).
Hook session_before_compact is emitted with the preparation (messagesToSummarize, firstKeptEntryId, tokensBefore, previousSummary, fileOps, settings). Extensions may cancel or supply their own summary.
Summarize If no override arrived, the session calls generateSummary() with the structured summary prompt (Goal, Constraints & Preferences, Progress, Key Decisions, Next Steps, Critical Context) and the previous summary when this is a re-compaction.
Append A new CompactionEntry is appended to the JSONL session file. It carries summary, firstKeptEntryId, tokensBefore, and details: { readFiles, modifiedFiles }.
Reload session_compact is emitted, the session rebuilds itself from firstKeptEntryId onward, and the LLM next sees system + summary + recent tail instead of the full history.

Cut-point rules

  • Valid cut sites: user messages, assistant messages, bash executions, custom messages.
  • Tool results stay with their tool call вҖ” the cut never lands between them.
  • Turns start at user messages (or bash executions) and end at the next one.
  • Re-compaction walks back to the previous CompactionEntry.firstKeptEntryId, so messages that survived the earlier compaction are folded into the next summary.

Split turns

If a single turn alone exceeds keepRecentTokens, the cut lands mid-turn on an assistant message. The summary is then generated in two pieces вҖ” a history summary and a turn-prefix summary вҖ” and merged.

Serialization before summary

Before the LLM is asked to summarize, messages are flattened to text with serializeConversation(): 

[User]: What they said
[Assistant thinking]: Internal reasoning
[Assistant]: Response text
[Assistant tool calls]: read(path="foo.ts"); edit(path="bar.ts", ...)
[Tool result]: Output from tool

Tool results are truncated to 2000 characters during serialization; anything beyond that becomes a marker noting how many characters were dropped. This is what keeps the summarization request itself from blowing the context window.

Summary format

What the summary actually contains

The LLM is prompted to return the exact Markdown structure below. The harness then writes it verbatim into CompactionEntry.summary. 

## Goal
[What the user is trying to accomplish]

## Constraints & Preferences
- [Requirements mentioned by user]

## Progress
### Done
- [x] [Completed tasks]

### In Progress
- [ ] [Current work]

### Blocked
- [Issues, if any]

## Key Decisions
- **[Decision]**: [Rationale]

## Next Steps
1. [What should happen next]

## Critical Context
- [Data needed to continue]

<read-files>
path/to/file1.ts
path/to/file2.ts
</read-files>

<modified-files>
path/to/changed.ts
</modified-files>

The <read-files> and <modified-files> blocks are extracted by utils.js and stored in CompactionEntry.details so file tracking accumulates across re-compaction. This is the same shape used for branch summaries.

Kimchi integration

What Kimchi adds on top

Two integration points and one advertisement:

1 В· Slash-command advertisement src/extensions/help.ts lists /compact as { kind: "entry", key: "/compact", desc: "Compact context window" }. The command itself is registered upstream; Kimchi only surfaces it in /help.
2 В· PreCompact hook delivery Upstream fires session_before_compact with { preparation, branchEntries, customInstructions, signal }. In src/extensions/hook-adapters/adapter.ts Kimchi's runPreCompact() translates that into the Claude Code PreCompact hook payload: 
{
  "trigger": "manual" | "auto",     // customInstructions ? manual : auto
  "custom_instructions": string | undefined
}
Any additionalContext returned by the hook is sent to the model as a non-blocking steering message.
3 В· PostCompact + session restart After the summary is written, upstream fires session_compact. Kimchi's runPostCompact() delivers PostCompact with { trigger } ("manual" if the summary came from an extension, "auto" otherwise), then immediately fires SessionStart with source: "compact" so the user's Claude Code SessionStart scripts re-run on the rebuilt session. Any returned additionalContext is queued as the first message of the next turn via deliverAs: "nextTurn".

Wiring (from src/extensions/hook-adapters/adapter.ts)

pi.on("session_before_compact", (event, ctx) =>
    runPreCompact(definition, pi, event, ctx))

pi.on("session_compact", async (event, ctx) => {
    await runPostCompact(definition, pi, event, ctx)
    await runSessionStart(
        definition,
        pi,
        { ...event, type: "session_compact" },
        ctx,
        pendingSystemPrompt,
    )
})

For Claude Code compatibility. The extensions/claude-code-hook-adapter resource reuses this same plumbing to translate pi's session_before_compact / session_compact events into Claude Code's PreCompact / PostCompact shell hooks, so existing Claude Code hook scripts work without changes.

Caveats observed in kimchi code.

  • src/extensions/model-guard.ts falls back to local token estimation when getContextUsage returns null, which is exactly the post-compaction state вҖ” so the guard re-arms immediately after a compaction without a fresh API call.
  • benchmark/terminal-bench-2/src/kimchi_agent/claude_code_kimchi.py sets CLAUDE_CODE_AUTO_COMPACT_PERCENT = 85 when wrapping Claude Code in benchmark mode вҖ” that is a Claude Code-side override, not a kimchi CLI override, and only affects the benchmark harness.
Failure modes

What can go wrong

Symptom Cause What happens
Compaction never fires even on long sessions compaction.enabled: false in settings.json shouldCompact short-circuits to false. Overflow recovery also returns early. Manual /compact still works.
Session still overflows mid-turn Single turn larger than keepRecentTokens + reserve The cut lands mid-turn, generating a split-turn summary. If the summary itself is too large, the next turn will retry through overflow recovery.
Summary is "wrong" / loses important context Lossy summarization is by design Full history stays in the JSONL session file. Use /tree or fork the session to revisit it.
Extension cancels session_before_compact Hook returned { cancel: true } Compaction aborts. The proactive threshold will re-check on the next turn; overflow recovery will retry the failing turn without compaction.
Compaction runs but the next turn fails immediately Summary itself is too large, or the LLM rejected the structured prompt Treat as overflow вҖ” upstream will run overflow recovery on the next failing turn and may re-compact.
