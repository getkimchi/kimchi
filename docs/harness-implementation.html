Kimchi internals

How the coding harness turns LLM calls into an agent

Kimchi is not a full agent loop reimplementation. It is a harness around @earendil-works/pi-coding-agent that changes startup configuration, model routing, system prompts, tool visibility, permission gates, subagent execution, and ferment project automation.

1

Upstream owns the loop

The upstream package creates sessions, builds provider calls, executes model turns, streams assistant output, runs tools, compacts context, and persists session history. Kimchi enters that loop by passing extension factories to upstream main() or by creating child sessions for subagents.

2

Kimchi owns steering

Before each model run, Kimchi assembles a custom system prompt from live environment data, project guidelines, skills, active tools, extension prompt blocks, phase guidance, and mode-specific orchestration instructions.

3

Modes change the contract

Standard sessions handle user tasks directly. Multi-model sessions force a configured orchestrator model and delegate explore, build, and review through the Agent tool. Ferment sessions add a host-owned project state machine and require step work to be performed by workers.

Boot sequence

Startup path

The entry point is intentionally split. src/entry.ts has no transitive pi-mono imports, so it can set environment variables before upstream configuration is loaded. Only after the harness directory, package resources, proxy handling, OAuth template path, title, and paste interceptor are installed does it dynamically import src/cli.ts.

entry.ts Sets PI_PACKAGE_DIR, redirects PI_CODING_AGENT_DIR to ~/.config/kimchi/harness, installs proxy and paste hooks.
cli.ts Handles Kimchi subcommands, telemetry, setup, model metadata refresh, themes, keybindings, and terminal probes.
extension stack Builds the ordered list of Kimchi extensions: auth, permissions, prompt enrichment, ferment, agents, MCP, web tools, telemetry, UI, and guards.
upstream main Calls @earendil-works/pi-coding-agent main() with Kimchi extensions, or runAcpMode() for IDE/ACP JSON-RPC mode.
session events Extensions respond to session_start, before_agent_start, context, tool_call, turn_end, and session_shutdown.
Why this matters: most agent behavior is not encoded in one loop. It emerges from extension hooks around upstream session execution. The critical hook for LLM-call steering is before_agent_start, where Kimchi returns a complete system prompt override.
LLM call construction

The system prompt pipeline

prompt-enrichment.ts registers a before_agent_start handler. For every agent run it gathers active tools, project context files, skills, git metadata, current date, current phase, model registry data, and session-scoped prompt blocks. It then calls buildSystemPrompt(). 

mode = subagent ? "subagent"
     : multiModelEnabled ? "orchestrator"
     : "single"

systemPrompt = buildSystemPrompt({
  tools: activeTools,
  env: cwd / docs dir / git / date,
  contextFiles: AGENTS.md and related context,
  skills: loaded skills,
  currentModelId,
  currentPhase,
  registry,
  mode,
  roles,
  sessionId
})

The prompt builder always starts with the Kimchi identity and environment, then adds the Documents policy, core coding guidelines, factual accuracy rules, orchestration instructions, per-model phase guidelines, project guidelines, extension prompt blocks, available tools, and skills.

Subagents get the same general prompt construction machinery, but buildSystemPrompt() filters out delegation tools such as Agent, get_subagent_result, and steer_subagent. That prevents recursive delegation loops in worker sessions.

Prompt ingredients

Steering blocks

Base instructions Defines Kimchi as a coding agent and tells the model to use only listed tools.
Documents policy Directs plans, specs, research notes, and agent handoff files to .kimchi/docs.
Mode instructions Single-model, orchestrator, and subagent mode each receive different rules.
System prompt blocks Extensions such as ferment and permissions register session-scoped blocks that can inject or suppress sections.
Tools and skills Only currently active tools are listed. Skills are loaded from configured Kimchi, vendor, package, and project skill paths.
Modes

Standard, single-model, multi-model, and ferment

Mode How it starts LLM contract Tool behavior
Standard session Default harness path through upstream main(). Normal coding agent session. It may be single-model or multi-model, depending on settings.json and explicit model flags. Uses the full active tool surface, subject to permissions, resource toggles, and model guards.
Single-model Explicit --model at startup or set_model to a concrete model disables multi-model. The prompt says all work runs on the selected model. If subagents are used, the model must pass its own model ID, not a different role model. Delegation remains possible for parallelism or isolation, but not for cross-model specialization.
Multi-model Default when multiModel is true and no explicit model flag is set, or via set_model({ model: "multi-model" }). The main model becomes the orchestrator. It classifies the task, chooses phases, and delegates explore/build/review according to configured role models. session_start force-switches the main session back to the configured orchestrator model if the user selected another model.
Ferment First-run session picker, /ferment, /ferment new, or /ferment one-shot. The main model acts as a ferment planner. It scopes a project, drives a persistent FSM, and must delegate implementation steps to workers. Ferment lifecycle tools become visible according to runtime profile. Active ferment switches permissions toward autonomous execution unless a CLI mode overrides it.
Plan permissions mode --plan, /permissions mode plan, or questionnaire promotion. Not a model orchestration mode; it is a permission/tool mode. A prompt block steers the model to plan and emit a completion marker. Mutating tools are hidden or blocked. Read-only tools, web tools, questionnaire, and read-only bash remain available.
Multi-model orchestration

Role-based model routing

model-roles.ts defines six roles: orchestrator, planner, builder, reviewer, explorer, and judge. Defaults are OSS Kimchi models, but ~/.config/kimchi/harness/settings.json can override any role under modelRoles.

Role Default Used for
orchestrator kimchi-dev/kimi-k2.6 Main loop. Classifies tasks, plans/delegates, coordinates results.
planner kimchi-dev/kimi-k2.6 Writes specs when planner differs from orchestrator; otherwise orchestrator plans.
builder kimchi-dev/minimax-m2.7 Implementation, code edits, refactoring, and build chunks.
reviewer kimchi-dev/minimax-m2.7 Independent verification, lint/test checks, and review findings.
explorer kimchi-dev/nemotron-3-super-fp4 Codebase exploration, architecture tracing, and broad research.
judge kimchi-dev/kimi-k2.6 Ferment verification triage and final journey grading.

The orchestration prompt creates a Your Team section from these roles. Each role line includes display name, provider/model ID, description, and vision support when known. The model metadata comes from a Kimchi-specific ModelRegistry, built by merging live API model metadata with local capability metadata such as tier, strengths, vision, and description.

Important distinction: pi-mono's model registry handles provider config, API keys, and actual Model objects. Kimchi's orchestration registry is advisory metadata for routing and prompt generation.
Orchestrator prompt

What the main model is told to do

In multi-model mode, the orchestrator receives a strict phased pipeline. It must classify the task as simple or complex, select required steps from explore, research, plan, build, and review, and decide what it owns versus what must be delegated.

  • Explore: always delegated to the Explorer model.
  • Build: always delegated to the Builder model.
  • Review: always delegated to the Reviewer model.
  • Research: direct for a single web-search answer, delegated for long or multi-source work.
  • Plan: self-served when planner equals orchestrator; otherwise delegated to the Planner model.

For complex tasks the prompt requires a Markdown spec in the Documents directory, self-validation, optional external plan verification, chunked build delegation, and final independent review. It explicitly forbids the orchestrator from editing implementation files after delegating.

Subagent prompt

What workers are told to return

Subagents are started through Kimchi's Agent extension. The generic subagent mode prompt requires a final response that is exactly one JSON object: 

{"summary":"one paragraph","files":["/absolute/path/to/doc.md"]}

Substantive output such as specs, plans, research notes, and findings should be written to files in the Documents directory, not embedded in the final message. This keeps handoffs small and lets the orchestrator pass file paths between workers instead of bloating the conversation context.

Agent personas can further override worker behavior: allowed tools, memory, skills, extensions, model list, thinking level, max turns, token budget, and prompt mode are resolved before the child session is created.

Agent tool implementation

How subagents become additional LLM calls

Kimchi registers three LLM-callable delegation tools: Agent, get_subagent_result, and steer_subagent. The Agent tool does not call a remote service directly. It asks AgentManager to start a child AgentSession using upstream createAgentSession().

Tool call Model calls Agent with type, prompt, description, optional model, background flag, budget, turns, isolation, and context inheritance.
Invocation config Kimchi resolves persona settings, model choice, join mode, token budget, max turns, visibility, and session file.
Child prompt runAgent() builds the worker system prompt with parent context, environment, memory block, skills, guidelines, and persona instructions.
Child session Upstream createAgentSession() starts an in-memory or persisted child session with selected model and filtered tools.
Lifecycle control Kimchi tracks tool activity, streaming text, usage, compaction, turns, inactivity, duration, and abort reasons.

Background agents enter a queue with a default concurrency limit of four. The manager stores records with status, model ID, tool-use count, session, output, error, usage, and abort reason. Completed background agents notify the parent session, and the model can fetch full results later with get_subagent_result.

Steering is implemented through upstream session steering. Kimchi prefixes automated steering messages with an orchestrator marker so the worker does not interpret them as user input. Progress steering fires at 50%, 75%, and 90% of turn budget; hard aborts occur after grace turns, token budget overflow, inactivity, or max duration.

Tool surface

Tools available to the model

The final prompt lists only active tools. Upstream provides core file and shell tools, while Kimchi extends or wraps many of them.

read write edit bash grep find ls web_search web_fetch Agent get_subagent_result steer_subagent set_model set_phase questionnaire Skill curator MCP tools ferment lifecycle tools

MCP tools are registered through the MCP adapter when enabled. Tool rendering/grouping extensions customize how calls appear in the TUI but do not by themselves change model reasoning.

Tool gating

Permissions are part of orchestration

permissions/index.ts intercepts tool_call events. It resolves mode from runtime override, CLI flags, environment, and config. Then it applies tool visibility, built-in deny rules, user/project allow and deny rules, read-only checks, classifier calls, and UI prompts.

  • default: read-only tools pass; mutating or risky tools prompt when UI is available.
  • plan: mutating tools are hidden or blocked; bash must be read-only.
  • auto: non-read-only calls are classified by an LLM before approval or block.
  • yolo: bypasses permission checks and is marked dangerous.

Active ferment can runtime-switch permissions toward yolo so lifecycle work does not stall on repeated prompts. Explicit CLI flags still win.

Ferment

Autonomous project execution mode

Ferment is a higher-level workflow around the same upstream agent machinery. It stores a project plan in .kimchi/ferments/<uuid>.json, uses a deterministic state machine for valid transitions, injects planner instructions into the system prompt, and exposes lifecycle tools that the LLM must call to advance the plan.

Request The user starts ferment through the picker or slash commands. The host owns creation; the model is told not to create .kimchi files manually.
Scoping The planner calls propose_ferment_scoping with title, goal, success criteria, assumptions, phases, questions, and P1/P2/P3 gates.
Plan review The host displays a plan-review UI. Confirmation persists the plan and moves the ferment to planned.
Execution The planner activates/refines phases, starts steps, delegates each step to subagents, then completes steps with gate evidence.
Completion Phase and final completion tools validate gates, run project-check logic, record evidence, apply continuation policy, and optionally grade the journey.

Ferment prompt supplement

When an active ferment exists, buildFermentPromptBlock() injects a planner role into the system prompt. It tells the main model that it manages the task graph and must delegate all implementation work. It lists available subagent types, current scoping context, decisions, memories, manual/automated continuation policy, and detailed rules for questions, phase design, parallel steps, and gate payloads.

Ferment lifecycle tools

The planner-facing tool set is explicit: request_ferment_workflow, propose_ferment_scoping, list_ferments, scope_ferment, update_ferment_scope_field, complete_ferment, activate_ferment_phase, refine_ferment_phase, complete_ferment_phase, skip_ferment_phase, fail_ferment_phase, start_ferment_step, complete_ferment_step, verify_ferment_step, skip_ferment_step, fail_ferment_step, add_ferment_decision, add_ferment_memory, and ask_user.

Key design: the model is not trusted to mutate ferment state directly. Tool handlers validate UI gates, call the deterministic state machine, persist the result, run host-side side effects, and return the next action hint that drives the following model turn.
Ferment verification

Gates, evidence, and judge calls

Ferment completion tools require structured gate verdicts. Each gate includes an ID, verdict, rationale, and evidence. Validation enforces gate coverage and rejects empty or malformed evidence. Gate flags become block flags that feed retry and escalation logic.

The judge is intentionally narrow. Most grading was replaced by deterministic gate validation. The surviving LLM judge calls are tactical: interpreting non-zero step verification output and assigning a final journey grade after final completion gates have already passed.

Judge model selection uses the configured judge role when available, falling back to captured session model context.

Continuation

Manual vs automated ferment

Ferment has a continuation policy independent of standard permission mode. In manual policy, phase boundaries pause and ask the user whether to proceed. In automated policy, the planner continues across phase boundaries until the ferment completes, blocks, pauses, or needs user input.

Events such as turn_end, agent_end, and scheduled wake-ups deliver follow-up messages to keep the state machine moving without the user typing another prompt. Stuck-loop counters block repeated starts or repeated block hashes and redirect the planner toward recovery.

Runtime safeguards

How Kimchi keeps LLM calls from drifting

  • Continuation nudges: if the orchestrator announces delegation but emits no tool call, hidden follow-up messages nudge it to continue instead of ending the loop.
  • Empty-turn nudges: textless or malformed turns can be recovered by a hidden message rather than waiting for the user.
  • Context cleanup: stale nudges, UI-only messages, and empty-name tool calls are stripped from context before the next LLM call.
  • Model switching guard: model changes are rejected when the current context exceeds the target model's safe context window.
  • Vision guard: sessions containing images cannot switch to a non-vision model unless images are stripped or described.
  • Agent budgets: child agents can be limited by max turns, output-token budget, inactivity timeout, and wall-clock duration.
  • Permission fail-closed behavior: when prompts or classifier models are unavailable for risky calls, the tool call is blocked instead of implicitly allowed.
  • Plan persona path scope: plan subagents may write only under .kimchi/plans/.
