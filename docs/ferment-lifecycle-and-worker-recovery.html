01 · Mental model

The plan is durable state; the model is the driver

A Ferment is not a long prompt pretending to be a workflow. It is a persisted project record. A deterministic engine reads that record and recommends the next legal action; the planner performs the action through tools.

1

Persistent record

Goal, criteria, constraints, phases, steps, decisions, memories, and statuses live in .kimchi/ferments/<id>.json.

2

Deterministic engine

determineNextAction converts current state into one action such as scope, activate, refine, start, complete, recover, or finalize.

3

Planner orchestration

The planner follows lifecycle tool results, delegates implementation, and never silently mutates phase or step state.

4

Linked worker evidence

A step finishes only when the correct linked Agent has both a completed outcome and a completed structured report.

Core invariant: a next-action hint is guidance, not proof. In particular, complete_step is only a suggestion until worker identity, linkage, outcome, report, gates, and verification all permit the transition.

02 · Lifecycle overview

From rough intent to explicit completion

The outer Ferment owns phases; each phase owns steps; each running step owns delegated worker activity. The phase/step loop repeats until every phase is terminal, then complete_ferment finalizes the project.

Draft Intent, discovery, questions
→
Planned Goal and phases accepted
→
Active phase Refine into concrete steps
→
Running step Bounded linked Agent worker
→
Completed All phases terminal + explicit final tool
↺ Worker completes → step completes/verifies → next step starts → phase completes → next phase activates

Three nested state machines

Ferment
draft→planned→running→complete

paused is resumable. abandoned is terminal.

Phase
planned→active→completed

A phase may also become skipped or failed.

Step
pending→running→done

Terminal alternatives are verified, skipped, and failed.

03 · One step, end to end

The planner delegates; the worker reports; the tool validates

Implementation is deliberately split across three actors. This prevents the planner from treating a plausible summary as completion evidence or implementing a delegated step inline.

Moment
Planner
Ferment tools / state
Linked Agent worker
1 · start
Calls start_ferment_step
Step becomes running; returns exact task_ref + suggested limits
2 · spawn
Calls Agent with task_ref, max_turns, max_duration
Receives Agent ID, report token, step context, and report protocol
3 · work
Edits, tests, verifies, tracks completed and remaining work
4 · report
Secures report against Agent ID + nonce + task_ref
Calls submit_agent_report before final answer
5 · decide
Inspects agent_outcome and report
Returns completed, budget_exhausted, failed, or stopped
6 · complete
Calls complete_ferment_step with worker_agent_id
Validates linkage and evidence, runs gates/verification, then mutates state

Suggested worker bounds

Tier Typical signal max_turns max_duration
Light Read, check, verify, rename, lint, format, docs 25 300s
Standard Typical implementation without heavy build signals 50 600s
Heavy Compile, build, install, Docker, migration, Make/Cargo/Gradle 80 900s

These are starting suggestions, not permission to retry indefinitely. Both limits are always present so a worker cannot consume unbounded turns or hang forever on a blocking operation.

04 · Completion gate

“The worker returned” is not enough

complete_ferment_step validates the worker record before gate checks or state mutation. The report token also prevents another Agent from submitting progress on behalf of the linked worker.

Identity

Correct Agent?

  • worker_agent_id exists
  • Agent record is known
  • Not an unrelated/system worker
→
Linkage

Correct step?

  • task_ref.kind = ferment_step
  • Ferment ID matches
  • Phase ID and step ID match
→
Evidence

Actually complete?

  • Latest outcome is completed
  • Structured report exists
  • Report status is completed
Rejected states: missing report, partial/blocked report, budget exhaustion, failure, stop, missing outcome, wrong task reference, or unknown Agent ID. No rejected state can transition the step to done.

05 · Budget exhaustion recovery

Preserve useful work without claiming success

Exhaustion is a checkpoint, not completion. The planner reads the structured report and chooses the smallest recovery action that fits what remains.

Outcome

budget_exhausted

Keep the step running. Inspect report and remaining_steps.

Direct continuation

Resume the same Agent

Use a bounded fresh allocation and a steering prompt. Preserve its session context.

Stalled approach

One changed-approach resume

Keep the thread only when its context matters; explicitly change the attempted strategy.

Clean task boundary

Spawn a narrower linked replacement

Carry forward completed work and delegate only the separable remaining task with the same step task_ref.

Missing report only

Run a one-turn finalizer

Resume with max_turns: 1; forbid more task work; collect submit_agent_report.

Blocked / unclear

Stop and surface the blocker

Do not invent completion and do not let the planner implement the remaining work itself.

Anti-pattern

Never retry the same broad task

Do not simply raise limits. Narrow, steer, replace, or stop based on evidence.

Only one path reaches completion: a linked worker eventually returns outcome = completed with report.status = completed, after which normal step gates and verification still apply.

06 · Automated continuation

Two nudges recover two different stall patterns

The incoming benchmark-driven changes made automated Ferments more persistent without creating an infinite nudge loop. Every nudge is state-derived and capped.

Reactive nudge

Text-only early stop

The assistant responds without a lifecycle tool call while state still has a next action. Kimchi injects one contextual follow-up, then suppresses repeated nudges.

Stop nudge

Tool call, summary, then stop

The assistant made progress but ended the turn before the next lifecycle transition. Kimchi injects an action-specific follow-up, capped at two consecutive early-stop turns.

Contextual action

No generic “continue” prompt

The scheduler names the concrete phase, step, IDs, verification command, and next lifecycle tool derived from persisted state.

Safety boundary

Pause and terminal states win

Paused, complete, and abandoned Ferments do not receive continuation nudges. Manual policy still stops at phase boundaries.

07 · What changed

Incoming persistence + our completion integrity

The merge combined two complementary goals. Incoming code reduces premature stops and runaway work. Our branch makes worker progress resumable, attributable, and impossible to misclassify as completion.

08 · Operational invariants

Rules that must remain true

1

No unbounded worker

Every Ferment Agent call carries both turn and duration limits.

2

No anonymous completion

Every step completion names the linked worker Agent ID.

3

No aborted completion

Budget exhaustion, failure, and stop outcomes keep the step incomplete.

4

No missing report

A completed outcome without a completed report requires a finalizer resume.

5

No broad blind retry

Recovery must steer, narrow, replace, or stop based on remaining work.

6

No silent lifecycle mutation

The model calls lifecycle tools; persisted state remains the source of truth.

09 · Implementation map

Where each behavior lives

The behavior is intentionally distributed by responsibility: state selection, tool enforcement, worker runtime, and prompt-level orchestration are separate layers.

src/ferment/engine.ts
Priority-ordered, declarative next-action selection from persisted state.
src/ferment/state-machine.ts
Legal state transitions and mutation rules for Ferments, phases, and steps.
src/extensions/ferment/tools/steps.ts
Worker limit suggestions, exact task_ref handoff, linked-worker completion validation, gates, and verification.
src/extensions/agents/manager/agent-manager.ts
Worker resumption, Agent outcome tracking, and injected structured-report protocol.
src/extensions/agents/index.ts
Secured submit_agent_report tool, report validation, and recovery guidance.
src/extensions/ferment/prompt-block.ts
Planner invariants: delegation, bounded workers, outcome-aware recovery, and turn discipline.
src/extensions/ferment/oneshot.ts
One-shot parity with the same worker linkage and completion rules.
src/extensions/ferment/nudge.ts
Reactive text-only and post-tool early-stop continuation recovery with capped counters.
src/extensions/ferment/scheduler.ts
Context-rich action nudges that name concrete phases, steps, IDs, and verification commands.
src/extensions/bash-default-timeout.ts
Default 120-second upper bound for bash calls that omit an explicit timeout.
