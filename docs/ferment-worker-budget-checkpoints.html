Ferment Worker Budget Checkpoints

This document explains how Kimchi's Ferment planner and Agent workers interacted before the budget checkpoint change, how they interact now, and why the new contract prevents incomplete worker output from being treated as completed Ferment step work.

Audience And Mental Model

Kimchi extends the Pi coding agent with local extension code. The upstream agent loop still lives in @earendil-works/pi-coding-agent. The code described here is Kimchi-owned extension behavior in src/extensions/agents/, src/extensions/ferment/, and src/extensions/orchestration/.

Agent

The Agent tool starts a child coding session. It can run foreground or background, use a specialized persona, and enforce budgets such as max_turns, token_budget, max_duration, and inactivity timeout.

Ferment

Ferment is a task workflow engine. A Ferment has phases and steps. The planner is supposed to manage the state machine, while implementation work for each step is delegated to Agent workers.

Orchestrator

The primary model session coordinates work. It calls Ferment tools, starts Agent workers, waits for results, and decides whether to complete, resume, replace, skip, or stop a step.

System map

Orchestrator

The main model turn. It decides when to start Ferment steps, spawn Agents, resume workers, and complete or stop work.

->
Ferment state machine

Owns phases, steps, legal transitions, gates, and the rule that a step can advance only with valid evidence.

->
Agent worker

Executes delegated work in a child session, enforces budgets, and reports an agent_outcome.

Previous Behavior

Before this change, Ferment and Agent both existed, but their contract was loose. Agent knew whether a worker stopped because of a budget or timeout. Ferment knew which step was running. The missing piece was a stable, machine-checkable link between "this worker" and "this Ferment step", plus a model-visible distinction between successful completion and budget exhaustion.

1. Start step start_ferment_step marked the step as running and told the planner to spawn a subagent.
2. Spawn worker The planner called Agent. Budgets could be supplied, but the Ferment step was not formally linked to the Agent record.
3. Read result The Agent result was mostly free-form text plus legacy runtime status fields. Budget exhaustion looked like a kind of aborted textual result.
4. Complete step The planner could call complete_ferment_step with a summary and gates. Ferment did not verify that the claimed worker was linked, completed, or even present.
Main failure mode: if a worker hit max_turns or token_budget, the orchestrator might still interpret its partial checkpoint text as enough evidence and complete the Ferment step. That made budget exhaustion look too much like normal completion.

Previous flow

start_ferment_step marks the step running
v
Planner spawns Agent, usually with a free-form prompt
v
Agent returns text; budget exhaustion can be buried in the prose
v
Planner may call complete_ferment_step even though worker did not truly complete

Current Behavior

The current design makes Agent outcomes structured and resumable, then makes Ferment step completion depend on a completed linked worker. Budget exhaustion is now an explicit checkpoint, not a successful result.

1. Start step start_ferment_step returns the exact task_ref the planner must pass to the worker.
2. Spawn worker The planner calls Agent with max_turns and task_ref. The worker is told to produce a checkpoint if it nears the limit.
3. Agent records outcome Agent stores latestOutcome on the AgentRecord and exposes an agent_outcome JSON block in foreground and background results.
4. Complete step complete_ferment_step requires worker_agent_id and accepts only linked workers whose latest outcome is completed.

Current flow

Accepted path

start_ferment_step returns task_ref
v
Agent runs with task_ref + max_turns
v
agent_outcome.outcome = completed
v
complete_ferment_step accepts worker_agent_id, then gates run

Checkpoint path

start_ferment_step returns task_ref
v
Agent reaches max_turns or token_budget
v
agent_outcome.outcome = budget_exhausted
v
complete_ferment_step rejects; planner must resume or replace worker

The Agent Outcome Contract

Agent keeps old status values for UI and backward compatibility, but adds a more specific outcome value for orchestration decisions.

Runtime status New outcome Meaning for Ferment
completed or steered completed The worker finished acceptably. Ferment may complete the linked step if gates also pass.
aborted with max_turns or token_budget budget_exhausted The worker produced a checkpoint, not a completion. Resume or replace it before completing the step.
error or other failed abort reasons failed The worker failed. Spawn a corrected replacement or stop and report the failure.
stopped stopped The worker was stopped. It is not valid completion evidence.

Task References

A task_ref is a lightweight link from an Agent worker back to the Ferment step it is working on. It is not a general task graph. It is just enough identity for Ferment to answer: "Does this Agent belong to the step that is being completed?" 

{
  "kind": "ferment_step",
  "ferment_id": "019edefb-da33-70ce-8a80-7a8b5e94088d",
  "phase_id": "phase-1",
  "step_id": "step-1"
}

start_ferment_step now includes this object in its next-action hint. The planner should pass it unchanged into the Agent call. 

Agent({
  subagent_type: "Builder",
  model: "provider/model-id",
  description: "implement step",
  max_turns: 12,
  task_ref: {
    kind: "ferment_step",
    ferment_id: "019edefb-da33-70ce-8a80-7a8b5e94088d",
    phase_id: "phase-1",
    step_id: "step-1"
  },
  prompt: "Implement this Ferment step. If you approach max_turns, produce a checkpoint..."
})

Completion Validation

complete_ferment_step now performs worker validation before gate validation or any state mutation. This order prevents a bad worker result from changing Ferment state.

complete_ferment_step decision tree

Planner calls complete_ferment_step with worker_agent_id
Missing or unknown worker?
Reject provide or retrieve the correct Agent ID.
task_ref mismatches this Ferment step?
Reject spawn a worker linked to this exact step.
No latest outcome yet?
Wait use get_subagent_result, then retry completion.
outcome = budget_exhausted?
Checkpoint bounded resume or narrower replacement worker.
outcome = failed or stopped?
Reject corrected replacement or report failure.
outcome = completed?
Continue run gate validation and verification.
Input to complete_ferment_step Result
No worker_agent_id Rejected. The planner must provide the Agent ID returned by the worker.
Unknown Agent ID Rejected. The planner should retrieve the correct background worker or spawn a linked worker.
Agent exists but task_ref does not match the Ferment step Rejected. The planner must not use another step's worker as evidence.
Linked Agent has no recorded outcome Rejected. The planner should wait for the worker result, usually via get_subagent_result.
Linked Agent outcome is budget_exhausted Rejected with guidance to resume the same Agent or spawn a narrower linked replacement worker.
Linked Agent outcome is failed or stopped Rejected. The planner should spawn a corrected replacement worker or report the failure.
Linked Agent outcome is completed Accepted, then normal Ferment gates and verification run.

Resume Semantics

Agent({ resume: "<agent_id>", prompt, max_turns }) continues the same child session with a fresh per-resume turn allocation. It does not erase the old session. It records a resume attempt on the same AgentRecord.

Resume cycle

Worker returns budget_exhausted with checkpoint and remaining_work
v
Orchestrator chooses bounded resume or narrower replacement worker
v
Resume uses same Agent session plus fresh max_turns window
v
If resumed outcome is completed, Ferment can complete the linked step
  • max_turns is the primary Ferment worker budget.
  • max_duration remains a safety guard for hangs and blocking operations.
  • token_budget is optional and caller-specific.
  • By default, an Agent record allows at most two bounded resumes.
  • A resume returns the same structured agent_outcome shape as a fresh run.
Important: resuming is allowed, but blind retry is not. The resume prompt should use the checkpoint and ask for only the remaining work or a finalizer pass.

What The Orchestrator Should Do Now

  1. Call start_ferment_step.
  2. Copy the returned task_ref.
  3. Start an Agent with task_ref and a task-sized max_turns.
  4. Wait for or retrieve the worker result.
  5. Inspect agent_outcome.outcome.
  6. If it is completed, call complete_ferment_step with worker_agent_id.
  7. If it is budget_exhausted, resume the same worker with a bounded turn allocation, or spawn a new linked worker scoped to remaining work.
  8. If it is failed or stopped, spawn a corrected replacement or report the failure.

Why This Is Local Extension Code

The underlying Pi agent loop already provides session execution, extension hooks, and tool runtime behavior. This change is Kimchi-specific orchestration policy: how Ferment planners should treat Agent worker budgets, how workers are linked to Ferment steps, and when a Ferment step is allowed to advance. That makes it a local extension-layer behavior rather than an upstream patch.

Quick Debug Checklist

  • If complete_ferment_step says worker_agent_id is missing, pass the Agent ID from the worker result.
  • If it says the worker is not linked, check that task_ref matches the exact Ferment, phase, and step IDs.
  • If it says the worker exhausted budget, use the checkpoint and resume or replace the worker.
  • If the worker finished in the background, use get_subagent_result before completing the step.
  • If tests call completeStep directly, they need a fake linked Agent record or must assert the missing-worker error.
