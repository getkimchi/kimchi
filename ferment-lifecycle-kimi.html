🍺 Ferment Lifecycle

Complete technical reference for Kimchi’s autonomous project-execution mode

Philosophy States Lifecycle Engine State Machine Persistence Scoping Execution Verification Continuation Parallel Recovery Knowledge Tool Profiles Commands Files

1. Core Philosophy

“The plan IS the state.” A ferment is a single JSON file (.kimchi/ferments/<uuid>.json). The harness reads it at session start, a deterministic engine decides what to do next, and every mutation is immediately persisted. If the session crashes or ends, the next session resumes from exactly the same point.

Three-layer architecture
  1. Tool handlers (src/extensions/ferment/tools/) — validate UI gates, run side effects (judge, verification), format LLM-facing text.
  2. Pure state machine (src/ferment/state-machine.ts) — (ferment, command, ctx) → next ferment; no I/O, no time, no randomness.
  3. Storage / event store (src/ferment/store.ts, event-store.ts) — atomic read/write and an append-only event log.

2. Status Taxonomy

2.1 Ferment status

StatusMeaningCan mutate?
draftCreated; scoping in progress.Yes, via scoping tools.
plannedScoping confirmed; no phase activated yet.Yes.
runningAt least one phase is active.Yes.
pausedUser-intervention/session state; most tools blocked.Only resume / abandon.
completeExplicitly finalized after all phases terminal.No.
abandonedPermanently stopped.No.

2.2 Phase status

planned → active → completed / skipped / failed. A failed phase can be retried (activate_ferment_phase) or bypassed (skip_ferment_phase).

2.3 Step status

pending → running → done / verified / skipped / failed. failed steps can be recovered by starting them again. done means completed without a passing verification command; verified means the verification command returned exit code 0.

4. The Forward Engine (determineNextAction)

The engine in src/ferment/engine.ts reads canonical ferment state and returns the next action the LLM should take. It is priority-ordered:

PriorityConditionAction
0Status is complete or abandonedundefined (idle)
1No phases definedscope
2Status is pausedpause
3Any phase failedrecover_phase
4All phases terminalcomplete_ferment
5No active phase, status plannedactivate_phase (first planned)
6Status running but no active phasepause (recovered state)
7Active phase has no stepsrefine
8Active step failedrecover_step
9Step pendingstart_step
10Step runningcomplete_step (suggestion)
11All steps terminalcomplete_phase
The engine returns a DeclarativeAction (kind + IDs + reason). The scheduler turns that into a context-rich nudge naming the exact tool, IDs, and description/verify command.

5. Pure State Machine (applyCommand)

src/ferment/state-machine.ts contains pure transition logic. Every command either returns the next Ferment or a typed TransitionError with a code such as FERMENT_NOT_IN_STATUS, PHASE_NOT_IN_STATUS, STEP_RUNNING, or CONCURRENT_NON_PARALLEL_STEP.

5.1 Key invariants enforced

  • Only one non-parallel step may run at a time.
  • A phase cannot be refined while a step inside it is running (would destroy in-flight work).
  • complete_ferment requires every phase to be terminal.
  • Pausing resets running steps back to pending (their subagent is killed).
  • Singleton parallel_group values collapse to sequential; only cohorts of size ≥2 are parallel.

5.2 Lifecycle helpers

src/ferment/lifecycle.ts provides helpers such as activateSinglePhase (deactivates any other active phase first) and settleAfterPhaseTerminalPatch (sets status back to planned or running based on remaining active phases).

6. Persistence & Event Store

6.1 Legacy snapshot

FermentStorage (src/ferment/store.ts) reads/writes .kimchi/ferments/<uuid>.json. It supports migration from V1→V2→V3→V4, caches ferments in memory, resolves by ID prefix or fuzzy name, and writes atomically via temp-file + rename.

6.2 Event log (source of truth)

The new event-store model (src/ferment/event-store.ts) maintains an append-only <uuid>.events.jsonl file. Every state mutation produces one or more typed events (e.g. phase_activated, step_started, step_completed, ferment_completed). Each event carries preStateHash / postStateHash for tamper detection.

6.3 Command → events

src/ferment/event-mapper.ts translates each successful Command into the events appended to the log. It folds each emitted event back onto a cursor so the final hash matches the real fold.

6.4 File layout

.kimchi/ferments/
  <uuid>.json              ← snapshot cache (legacy format)
  <uuid>.events.jsonl      ← append-only event log
  <uuid>.phases.json       ← immutable phase snapshots
  <uuid>.stats.json        ← computed stats cache

6.5 Directory resolution

  1. KIMCHI_FERMENTS_DIR environment variable.
  2. <project-root>/.kimchi/ferments when inside a git/package project.
  3. ~/.config/kimchi/ferments global fallback.

7.3 Gate validation

Scoping requires P1, P2, P3 plan-scope gates. Each gate needs id, verdict (pass/flag/omitted), rationale, and evidence.

8. Execution Flow

8.1 Overview

draft → planned → running
            ↓
    activate_ferment_phase(phase-1)
            ↓
    refine_ferment_phase(phase-1, steps)
            ↓
    start_ferment_step(phase-1, step-1)
            ↓
    Agent worker executes step
            ↓
    complete_ferment_step(phase-1, step-1)
            ↓
    ... next step ...
            ↓
    complete_ferment_phase(phase-1)
            ↓
    activate_ferment_phase(phase-2) or pause on manual policy

8.2 Activating a phase

activate_ferment_phase transitions a planned (or failed) phase to active. If a different phase was active, it is demoted to planned. For parallel cohorts, activate_phase_group atomically activates every phase sharing the same groupIndex.

8.3 Refining a phase

refine_ferment_phase populates steps[]. Each step has a description and optional verify command. Steps can also belong to a parallel_group. Refine is rejected if any step in the phase is running.

8.4 Starting a step

start_ferment_step marks a step running. It records the git HEAD ref, emits a plan-first preamble, provides worker context (decisions, memories, prior step summaries), and returns parallel siblings if the step is part of a cohort.

8.5 Completing a step

complete_ferment_step:

  1. Validates step-level gates; agent-emitted flag verdicts block the call.
  2. If the step has a verify command, runs it (60s timeout, 2 retries by default).
  3. On exit 0, status becomes verified; otherwise done.
  4. A judge (Claude Opus 4.6) classifies the result and assigns a letter grade A–F.
  5. Records a pending compaction so the session can be compacted at agent_end.

8.6 Completing a phase

complete_ferment_phase:

  1. Validates phase-level gates P1/P2/P3.
  2. Runs deterministic project-check validation (tests, lint, typecheck are discovered but not executed).
  3. Gathers git evidence (diff, files changed) since the phase started.
  4. Combines project-check flags and gate-verdict flags; block flags produce grade F, warn flags produce B.
  5. Persists review evidence and escalates after repeated block retries.
  6. Applies continuation policy (manual = pause at phase boundary, automated = continue).

9. Verification & Grading

9.1 Verification command

Each step may specify a bash command. When the step completes, the harness executes it. The result (exit code, stdout, stderr) is stored in StepResult. A non-zero exit does not automatically fail the step; instead the judge decides whether to pass, retry, or fail.

9.2 Grades

GradeMeaning
AExcellent — objective met.
BGood — minor gaps / warn flags.
CAcceptable — significant gaps.
DPoor — objective barely met.
FFailed — objective not met / block flags.

9.3 Grade propagation

Step grades feed into phase grades; phase grades feed into the final ferment grade at complete_ferment. Grades are shown in the dashboard and injected back into the planner context.

10. Continuation Policy

The runtime continuation policy decides whether the engine’s next action is executed automatically or waits for user input.

PolicyBehavior
manualAgent asks before moving to the next phase. Work inside the current phase continues autonomously.
automatedKeeps going through all stages until complete, blocked, paused, or failed.

Controlled by /ferment manual and /ferment auto.

10.1 Phase boundary detection

src/extensions/ferment/continuation.ts detects a manual phase boundary when the next action is activate_phase and at least one earlier phase is already terminal. The UI shows a dropdown: “Continue to next phase / Pause here / Let me say something.”

11. Parallel Execution

Phases or steps sharing the same parallel_group number execute concurrently. Singleton groups collapse back to sequential to avoid mis-signaling the engine and UI.

Phase 3: "Backend API"    parallel_group: 1
Phase 4: "Frontend UI"    parallel_group: 1
Phase 5: "Deploy"         (sequential, after 3 & 4)

When starting a parallel step, the tool result lists parallel_siblings and instructs the agent to launch their Agents concurrently.

Set KIMCHI_FERMENT_DISABLE_PARALLEL=1 to globally collapse all parallel cohorts into sequential execution for debugging.

12. Recovery & Protection

12.1 Stuck-loop protection

If start_ferment_step is called on the same step 3 or more times without a matching complete_ferment_step, the tool blocks further starts and prompts the user: Retry / Skip step / Pause ferment.

12.2 Failed phase recovery

When a phase fails, the engine emits recover_phase. Options:

  • activate_ferment_phase — retry the phase.
  • skip_ferment_phase — bypass it.
  • /ferment abandon — stop the ferment permanently.

12.3 Pause / resume

/ferment pause persists status paused and resets running steps to pending. Most ferment tools are structurally blocked while paused. /ferment resume returns to running or planned based on whether an active phase exists.

12.4 Exit

/ferment exit detaches the active ferment from the session without deleting it. Planned/running ferments are paused first; completed/abandoned ferments are simply detached.

14. Tool Profiles

Ferment uses session profiles keyed on lifecycle state. pi-mono snapshots tools at turn start, so the profile unlocks on the next model turn after a transition.

ProfileStateTools available
idleNo active fermentAll non-ferment tools + list_ferments, request_ferment_workflow.
planningFerment exists, all phases plannedResearch tools + ferment planning tools. No bash/edit/Agent.
implementationAt least one phase activatedFull toolset including bash, edit, write, Agent, and all lifecycle tools.
workerSubagent context (KIMCHI_SUBAGENT=1)Empty at ferment level; agent manager decides worker toolset.

17. Lifecycle at a Glance

1. Create      → draft
2. Scope       → planned
3. Activate    → running
4. Refine      → phase has steps
5. Start       → step running
6. Work        → Agent executes
7. Complete    → step done/verified (or failed/skipped)
8. Phase done  → complete_phase
9. Loop        → next phase or manual pause
10. Finish     → complete_ferment
Every transition is persisted immediately, every mutation is an append-only event, and the engine always knows the next action from the current state alone.
